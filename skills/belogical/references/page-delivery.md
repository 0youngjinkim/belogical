# page-delivery — Page and screen delivery

> Role: lay the finished pyramid out on the page or screen so the reader can see it and grasp it fast.
> Loaded on: document (step 4). For presentation decks, apply section 5.
> Minto's delivery methods, distilled for markdown and slide environments. Numeric thresholds are recommended defaults.

## 1. The 30-second rule

Within 30 seconds of starting to read, the reader should hold the whole thought — introduction, apex, and Key Line points. Summarize the Key Line points up front and everything that follows becomes elaboration; nothing important ambushes the reader later. The 30 seconds is a judgment call, not a stopwatch — if the gist doesn't land within the first screenful, rewrite.

## 2. Headings carry messages, not categories

Category labels like 'Background', 'Conclusion', 'Findings' contribute nothing to comprehension. Put the section's core idea in the heading — but its job is to remind, so don't cram in the whole summary sentence; move reasons and conditional clauses into the body's first sentence. The exception is reference documents (READMEs, manuals) where navigational headings have hardened into convention (Installation, Usage) — readers find sections by those names, so keep them; the no-labels rule applies to sections that carry ideas.

Recommended rules for hierarchical headings:

- At least two headings per level — a lone child makes the summary relation with its parent false.
- Headings in one group take parallel form (if the first ends in a verb, so do the rest).
- A heading is not part of the running text — the body must read smoothly with the heading deleted.
- Introduce a group of subheadings in the parent's body text first.
- Add headings only where truly needed — only where they genuinely ease the reader's way.

The check: delete the body, keep only the headings, and read. If the headings alone rebuild the whole thought and their order already tells the story, it passes. If category-label headings or skipped steps show up, that's a structural defect, not a wording one — go back to the structure. Run this check before writing prose: reordering an outline is cheap, reordering finished pages is expensive.

## 3. Reinforce the structure visually

The original's five devices (hierarchical headings, underlining, numbering, indentation, bullet points) map in markdown to heading levels, bold, numbered lists, indentation and quotes, and bullets. The purpose is not looks but making the hierarchy visible — place children directly where they answer the parent's question, and give same-level items the same form. If a point sentence you've set off for emphasis runs long or stacks several subjects and predicates, fix the sentence before styling it.

## 4. Bridge groups with content

Structural narration ('this chapter has examined...; the next chapter will...') adds no information. Bridge with content: let the next section's first sentence pick up the previous section's key word. Close a long, complex section by integrating, not repeating. If the pyramid stands, a final conclusion section is theoretically unnecessary — if you include one, use it to prompt the reader's next action.

## 5. Presentations — the speaker is the show

A deck is not a report ported to slides. Slides are visual aids; separate what you will say from what you will show.

- One idea per slide. Title it with a complete sentence carrying a judgment ('Sales outlook is strong'), not a caption ('Sales outlook').
- Keep text short — about six lines / thirty words is the guideline; when it overflows, split the slide. Reduce numbers to a readable form ($4.9M rather than $4,876,987).
- Exhibits lead, text supports. Before choosing a chart's form, decide which of the five questions it answers — composition, comparison, change over time, distribution, correlation. Chart titles are also complete sentences carrying the answer.
- Storyboard order: write out the full introduction → note each slide's point → choose how to visualize it → script what you'll say → design → rehearse.
